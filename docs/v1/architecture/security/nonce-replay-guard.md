# Nonce and Replay Protection

OR3 Intern uses several mechanisms to prevent replay attacks and ensure one-time use of security tokens.

## Approval token consumption

Approval tokens are one-time use. When verified, `ConsumeApprovalToken` atomically marks the token as consumed. If the token was already consumed, verification fails with "already used or revoked." This prevents the same token from being used twice for the same or different actions.

Source: `internal/approval/tokens.go:54-61` (ConsumeApprovalToken)

## Ceremony one-time use

WebAuthn ceremonies (registration, login, step-up) are consumed once. `ConsumeWebAuthnCeremony` atomically marks the ceremony as consumed. If already consumed, it returns `ErrInvalidCeremony`. This prevents replay of the same authentication challenge.

Source: `internal/auth/service.go:504-528` (consumeCeremony)

## Pairing code exchange atomicity

When exchanging a pairing code for a device token, the broker uses compare-and-swap (`CompareAndSwapPairingRequestStatus`) to atomically change the request status from "approved" to "exchanged." If the swap fails (another exchange already happened), the operation is rejected. If the device token rotation fails after the swap, the code attempts to revert the status.

Source: `internal/approval/pairing.go:157-167`

## Skill run plan claiming

When a skill run is about to execute, `ClaimSkillRunPlan` atomically marks the plan as "running." If the claim fails (another worker already claimed it), the caller recovers the current state by loading the plan again.

Source: `internal/tools/skill_run.go:341-348` (runPreparedSkillRun)

## Request status protection

Approval requests and pairing requests both use atomic status transitions. Once a request moves from "pending" to any terminal state (approved, denied, canceled, expired), it cannot be moved again. The `resolveRequest` and `ResolveApprovalRequest` methods check current status before allowing transitions.

Source: `internal/approval/requests.go:12-29` (resolveRequest)
Source: `internal/approval/pairing.go:71-99` (resolvePairingRequest)

## Encrypted secret nonces

Each encrypted secret uses a new random nonce for AES-256-GCM. The same plaintext encrypted multiple times produces different ciphertexts. Nonces are generated by `crypto/rand`.

Source: `internal/security/store.go:298-325` (encryptBlob/decryptBlob)

## Token hashing

Sensitive tokens are never stored in plaintext:
- Device tokens stored as SHA-256 hashes
- Session tokens stored as SHA-256 hashes
- Pairing codes stored as SHA-256 hashes

Source: `internal/approval/devices.go:48-53` (token hash), `internal/auth/service.go:354-355` (session token hash), `internal/approval/pairing.go:62` (code hash)
