Audit Report: Doctor, Observability & Failure Diagnosis
🔴 HIGH — Potential Runtime Failures

1. Engine_provider only sees cfg.Provider.\* (legacy flat fields), never ProviderProfiles or ModelRouting
   internal/doctor/engine_provider.go:9-57 and config/readiness.go:131-157 — Both the doctor and readiness engine check cfg.Provider.APIBase, cfg.Provider.APIKey, cfg.Provider.Model, and cfg.Provider.EmbedModel. But with model routing (ProviderProfiles + ModelRoutingConfig), a user can configure providers entirely through the profiles map without touching the flat ProviderConfig. The doctor will report "missing" findings for fields that are perfectly configured through profiles. The syncLegacyProviderFromRouting (routing.go:153) back-fills these flat fields, but only save() calls normalizeProviderRouting first — during EvaluateReadiness and Evaluate on a loaded config, the fields should be populated. However, there's a subtle bug: readConfigFile (load.go:32) loads from JSON (which could have empty ProviderConfig if the user only set ProviderProfiles), then normalizeAndValidateConfig calls normalizeProviderRouting which populates the legacy fields. So this should usually work, but if someone creates a config by hand with only providers and modelRouting but no top-level provider block, the sync should happen. Let me re-examine: Load() → readConfigFile() returns defaults → normalizeAndValidateConfig() calls normalizeProviderRouting() which syncs back. So actually this should be fine on load. But EvaluateReadiness and Evaluate use the config as-is, and after normalizeProviderRouting, the fields should be set. Let me check more carefully...
   Actually looking at parseConfigFile (load.go:50-59), it unmarshals the file over the defaults. If the user config has providers (plural) and modelRouting but not provider (singular), the Provider field retains its default. Then normalizeProviderRouting at routing.go:45 checks if strings.TrimSpace(cfg.Provider.APIBase) != "" || strings.TrimSpace(cfg.Provider.APIKey) != "" before syncing — this means if the legacy fields are still empty (from defaults but user has no API key in env), the sync doesn't execute properly because it tries to infer the provider from APIBase but it's the default OpenAI one. Actually it does proceed because the default APIBase "https://api.openai.com/v1" is non-empty. So it will sync. This should be fine.
   Still, the doctor never checks cfg.Providers (the map) for configured providers. If someone has 3 providers but the primary chat provider (from model routing) has a key set, the doctor correctly checks cfg.Provider.APIKey which was synced. But the error messages say "AI provider key is missing" with no indication of which provider profile. This is a minor usability issue.
2. applyAutomaticFixes mutates config without re-running profile validation
   internal/doctor/fix.go:15-101 — ApplyAutomaticFixes applies fixes sequentially (e.g., changing Service.MaxCapability to "safe" and Service.SharedSecretRole to "service-client"). But it does NOT re-validate the profile after applying fixes. If a managed fix creates a new issue (unlikely for the current set, but fragile), it goes undetected. The doctor.go CLI re-evaluates after applying fixes, but ApplyAutomaticFixes itself does not.
3. validateConfigSnapshot writes to temp file then re-loads, but skips env overrides again
   internal/doctor/engine_config.go:50-65 — validateConfigSnapshot writes the current (potentially env-overridden and normalized in-memory) config to a temp file, then calls config.Load() on it. But Load() calls ApplyEnvOverrides again inside. If the original config had env overrides applied, the re-load would re-apply them, potentially duplicating or altering values. For example, if OR3_API_KEY is set and was already applied to cfg.Provider.APIKey, the re-load via Load(path) would apply it again — though this should be idempotent. However, if the in-memory config has different values than the env (e.g., the user changed the API key in the interactive TUI), the Load() call would override it with the env value again. This is a bug — the function should call normalizeAndValidateConfig directly or skip env re-application.
4. emit() function uses DefaultBuffer().Append() which has a race condition with tests
   internal/log/stdlib.go:56-65 — emit() acquires outputMu for stdout but calls DefaultBuffer().Append() outside that lock. Buffer.Append() (buffer.go:46) acquires b.mu internally, so there's no data race per se. But the sequence "buffer append then stdout write" is not atomic — a test that resets the buffer via ResetDefaultBufferForTest between the two operations would lose the log entry. Also, ResetDefaultBufferForTest (stdlib.go:23) replaces the global defaultBuffer without any synchronization, which IS a data race if any goroutine is concurrently calling emit().
   🟡 MEDIUM — Consistency & Integration Gaps
5. inferComponent heuristic is fragile — falls back to "service" for everything
   internal/log/buffer.go:159-174 — inferComponent tries to extract a component name from log messages by taking text before the first :. If no colon is found, or the prefix doesn't match componentNamePattern, it returns "service". Since all log messages from emit() (stdlib.go:64) are formatted as "component: message", component parsing should work for internal logs. But for the stdlib sink (stdlib.go:71-85), messages come from external log.Printf calls that may not follow the component: message format — all of those fall back to "service". This means all third-party or transitioned Go standard library logs are indistinguishable in the buffer.
6. inferLevel is keyword-based — misclassifies messages containing "error" or "warn"
   internal/log/buffer.go:176-186 — inferLevel scans messages for substrings like "error", "failed", "warn", "timeout". A message like "No errors found in the config" would be classified as LevelError. The stdlib sink uses inferLevel because it receives flattened strings without level metadata. Non-stringent classification is expected for this lossy path, but the heuristic will produce false positives.
7. Findings slice capacity is pre-allocated to 48 but ~15 engines may generate 3+ findings each
   internal/doctor/engine.go:16 — findings := make([]Finding, 0, 48). With ~15 finding engines each potentially generating 3-5 findings, 48 might be exceeded for complex configs (hosted profile + channels + MCP + webhook + profiles). While Go slices grow dynamically, the pre-allocation is slightly tight.
8. Doctor's severityFor vs severityForConfigureOrStartup inconsistency
   internal/doctor/engine.go:39-50 — These two functions handle ModeConfigurePostSave differently:

- severityFor checks blockOnStartup && (mode == ModeStartupChat || ModeStartupServe || ModeStartupService) — does NOT include ModeConfigurePostSave
- severityForConfigureOrStartup includes ModeConfigurePostSave, ModeStartupChat, ModeStartupServe, ModeStartupService
  This means findings using severityFor will never block during ModeConfigurePostSave, while ones using severityForConfigureOrStartup will. This is intentional but undocumented — the naming is unclear. Consider a comment or aligning the names.

9. Redacted message truncation can leak information thru truncation boundary
   internal/log/buffer.go:137,152-155 — Redaction runs BEFORE truncation (line 137 redact() then lines 152-155 runes[:2000]). If a secret spans the 2000-character boundary, part of it may be visible after truncation. Specifically, values matching token|secret|password|authorization=[^\s]+ would typically be caught by redaction before truncation, but edge cases where the key-value pair is near the boundary could be problematic. Low risk given the regex should catch inline tokens regardless of position.
10. AuditLogger.Record silently drops errors when key is empty
    internal/security/store.go:158-164 — When a.Key is empty but a.Strict is false, the audit event is silently dropped. This is by design (non-strict mode tolerates audit failures), but the caller receives no indication that the audit event was not recorded. The approval audit callers (audit.go:70) log the error from Record, but both audit() and AuditExecEvent discard the return value (\_ = b.audit(...)) when audit is unavailable.
11. doctor.go CLI re-evaluates report three times in fix mode
    cmd/or3-intern/doctor.go:81-116 — In --fix mode, the CLI calls Evaluate() up to 3 times (before fix, after automatic fix, after interactive fix). Each Evaluate() call runs ALL engine\_\*.go functions against the config. For a a typical config with no issues, this is $O$ findings × engines work repeated. While findings are pre-allocated and the work is fast (no I/O beyond config field checks), this is 3x unnecessary work. Could cache or lazily re-evaluate only changed areas.
12. providerReadinessIssues and providerFindings are separate implementations with same logic
    internal/config/readiness.go:131-157 vs internal/doctor/engine_provider.go:9-57 — Both files contain copy-paste logic for checking if APIBase/APIKey/Model are set. The readiness engine generates ReadinessIssue structs, while the doctor generates Finding structs. If either diverges (e.g., one checks cfg.Provider.Model while the other checks cfg.ModelRouting.Chat.Primary.Model), users get inconsistent diagnostics from or3-intern status vs or3-intern doctor.
13. approvalBrokerRequired doesn't check approvals.Enabled
    internal/doctor/engine_approvals.go:41-55 — approvalBrokerRequired iterates over all approval domain modes without checking if cfg.Security.Approvals.Enabled is true. The caller approvalFindings has an early return if !cfg.Security.Approvals.Enabled {return findings} before calling it. But if someone adds a new caller in the future, calling approvalBrokerRequired on a disabled approvals config would still flag that ask/allowlist modes need a key — misleading when approvals are disabled entirely.
    🟢 LOW — Code Quality & Maintenance
14. networkHasRemoteHTTPMCP in engine_network.go duplicates hasRemoteHTTPMCPServers in config/validate.go
    internal/doctor/engine_network.go:87-104 and internal/config/validate.go:555-573 — These two functions have identical logic (iterate enabled MCP servers, check for non-loopback HTTP URLs). If validation requirements change (e.g., adding a new transport type), both must be updated.
15. isLoopbackAddr duplicated across doctor/engine_network.go:70-85 and config/validate.go:547-553
    Same concern as #14 — the function exists in both packages with slightly different signatures (doctor's takes string, config's takes string). The doctor version is more thorough (parses with/without port, checks for empty/loopback). The config version only checks net.ParseIP().IsLoopback(). If someone adds IPv6 loopback support to one but not the other, they'll diverge.
16. log.buffer.go:153 uses []rune(message) for truncation — allocates heavily per truncated message
    Converting a string to []rune allocates a new backing array every time. For the 2000-char limit, this is minor (at most ~8KB per entry), but for high-throughput logging, this adds GC pressure. Could use utf8.ValidString check + slice on bytes for ASCII-only messages.
17. No structured/field-based logging helpers
    The log package only supports printf-style logging with component + message. There's Entry.Fields map[string]string in the struct, but it's never populated by the Debugf/Infof/Warnf/Errorf helpers. The Fields field exists only for entries added programmatically (e.g., from the stdlib sink or direct buffer.Append() calls). This means structured log consumers (like the service API) receive entries with empty fields.
18. Redact() regex has a minor bug for Bearer patterns
    internal/log/buffer.go:215-218 — The Bearer redaction replaces the entire match with "Bearer [redacted]", but the regex Bearer\s+[A-Za-z0-9._~+/=-]+ won't match if the token is preceded by something like "Bearer " in JSON. More importantly, the =-based redaction at line 219 splits on = which handles token=abc123 but NOT tokens passed as JSON keys like "token": "abc123". The = replacement on line 219 would turn the JSON "token": "abc123" into [redacted]": "abc123" since the split breaks on the key-value = that doesn't exist in JSON format.

---

Summary: The doctor engine is well-structured with 16 focused diagnostic engine files. The most impactful issues are #3 (temp-file re-validation reapplies env overrides, potentially masking issues), #4 (data race in ResetDefaultBufferForTest), and #17 (structured log fields defined but never populated). The duplication between config readiness checks and doctor findings (#12, #14, #15) is a maintenance tax — either should be the single source of truth. The logging buffer is adequate but has edge cases in heuristic inference (#5, #6) that limit diagnostic accuracy for third-party output.
