🔍 Quick Overview

Both products suffer from the same disease: they were built by brilliant engineers who exposed everything to the user. Every knob, every slider, every internal concept has found its way into the UI and CLI. The result is a system that requires a computer science degree to configure — and even then, you'll need to read 16+ markdown documents to understand it all.

HIGH The Core Problem

OR3 has two products (a terminal CLI and a mobile/web app) that expose the same 22+ configuration sections, 48+ env vars, and 30+ pages. A user needs to understand MCP, embeddings, subagents, runtime profiles, safety modes, channel policies, trusted origins, CIDRs, passkeys, and token budgets — just to get started.

NOTE What's Actually Good

  • The retro pixel-art aesthetic is warm and distinctive — it feels human
  • The plain-language layer (lang package) already maps technical errors to friendly messages — it just needs to be used more
  • The app's Simple Settings concept (8 themed sections) is the right direction — it just sprawled too far
  • The Electron First-Run Wizard is excellent — step-by-step, plain language, progress bar. The CLI needs this too.
  • Documentation is thorough (16+ docs) — just organized for engineers, not humans

😵 App-Specific Confusion

HIGH "Agent" is the Wrong Word

The bottom nav has an Agents tab. Non-technical users think of travel agents or insurance agents. The "Agent Command Center" uses "runner", "thinking level", "tool policy", "isolation" — pure developer jargon. A 90-year-old has no reference point for any of this.

→ Call it "Tasks" or "Delegation." Hide runners/models behind an "advanced" toggle.

HIGH No Onboarding for Web/Mobile Users

iOS/Android/web users land on an empty chat page. There's no guided setup. They must discover /settings/pair on their own. The "Hi, I'm or3-intern" greeting is friendly, but sending a message to an unpaired device just fails silently.

→ Show a 3-step onboarding card on first launch: (1) Connect to your computer (2) Pick a name (3) Try a chat.

HIGH Chat Composer Has Too Many Controls

The chat input has: mode switcher (Chat/Ask/Work/Admin), runner picker, model picker, thinking level slider, attachments, slash commands, file mentions, AND a send button. That's 7+ decisions before typing a single message.

→ Default to "Chat" mode with smart auto-selection. Put everything else under one "Options" button.

MEDIUM Terminal Exposed Without Warning

The Terminal page exposes a full Linux terminal via xterm.js. No safety warning. No confirmation. A curious tap opens a blinking cursor in a shell — terrifying for non-technical users, dangerous for everyone.

→ Gate behind a "I know what I'm doing" confirmation. Consider hiding entirely unless in "developer mode."

MEDIUM 30+ Page Routes is Absurd

Between Chat, Agents, Add, Computer (with sub-pages), Settings (with 8+ sub-pages), Activity, Approvals, Memory, Scheduled, Notes, Prompts, Pair, Passkeys, Heartbeat, Observability, Permissions, Skills, Addons — the navigation is a labyrinth.

→ Collapse to 4 tabs: Chat, Tasks, Files, Settings. Everything else is accessible from within those.

MEDIUM Settings Save Review Shows Raw Config Keys

When saving "Simple Settings" changes, the review screen shows internal config keys like routing.chatProvider, exec.allowedPrograms. This breaks the "simple" abstraction.

→ Show only the friendly label that changed, not the raw key.

MEDIUM Observability is a Developer Tool in User Settings

Live logs, traces, and service events are exposed at /settings/observability. This is pure debugging infrastructure that serves zero purpose for end users.

→ Move to a hidden Developer Mode or remove from navigation entirely.

😵 CLI-Specific Confusion

HIGH Four Competing Setup Wizards

New users face or3-intern setup, init, configure, AND settings — all doing similar things differently. The README literally says "Use setup for the plain-language path. Use settings when you want to revisit. Use configure from a normal terminal." That's cognitive overload.

→ ONE setup experience. or3-intern setup for first run, or3-intern settings to revisit. Deprecate init and configure from user-facing paths.

HIGH 19 Commands, 11 With Sub-Commands

The CLI has embeddings status|rebuild, skills list|info|check|search|install|update|remove, devices list|requests|approve|deny|rotate|revoke, pairing list|request|approve|deny|exchange, and more. The root help is a flat table — no grouping by user role.

→ Group commands: "Everyday" (chat, settings, version), "Setup" (setup, pair, doctor), "Advanced" (everything else). Hide advanced by default.

HIGH "Safety Mode" vs "Runtime Profile" — Two Systems, Same Job

The consumer layer uses "Relaxed / Balanced / Locked Down" safety modes. Underneath are 5 runtime profiles with different enforcement. The settings TUI exposes raw runtimeProfile values. Users don't know which abstraction to use.

→ Expose ONLY safety modes to users. Runtime profiles are an implementation detail.

MEDIUM Pairing Requires a Networking Degree

To pair a phone, you need: service running, correct listen address, trustedBrowserOrigins matching exactly, trustedBrowserCIDRs covering the client IP, shared secret, and optionally Tailscale. The Tailscale guide alone is 312 lines troubleshooting 5+ failure modes.

→ Add or3-intern connect-device --auto that detects network config and guides with copy-paste commands.

MEDIUM No "It's Working!" Moment

After setup, or3-intern chat works, but there's no celebration or next steps. The setup review says "Saved setup. OR3 is ready to chat." For a product that can do Telegram bots, Slack integrations, cron jobs, webhooks — users walk away with no idea what's possible.

→ Post-setup "What now?" with simple checkboxes: Try a chat, Connect Telegram, Pair your phone, Set up automation.

#5 — Three Settings, Not Twenty-Two

Reduce the 22 config sections + 8 Simple Settings groups to just 3 visible surfaces:

  • Safety — Relaxed / Balanced / Locked Down (one slider)
  • Connected Apps — Telegram, Discord, etc. (one card each)

  • Set a schedule — "I can check in on you daily"

#8 — Add Confirmation for Destructive Actions

"Disconnect this app," "Remove device," "Revoke passkey," "Delete scheduled task" — none have confirmation dialogs. Settings has a "last change undo" but it's not obvious. Add simple "Are you sure?" prompts with a 3-second grace period before the button activates.

or3-app

#9 — Show Only What's Configured

If a user hasn't set up Telegram, don't show the Telegram settings card. If they haven't enabled the service, don't show pairing options. If they haven't added passkeys, don't show passkey management. Progressive disclosure: show features as they become relevant.

Both Apps

#10 — Add a Dark Mode

It's 2026. The warm light theme is lovely, but dark mode is table stakes. The CSS custom property system is already in place — it's ready for it.

🧭 Simplification Principles to Live By

  1. If Grandma wouldn't understand the word, find a new word. "Embedding dimensions" → "Memory search." "MCP transport" → "Connection."
  2. The default path must work with zero configuration. Smart defaults for model, safety, workspace. User only provides API key.
  3. One way to do each thing. Not four setup wizards. Not three pairing methods. Pick the best one and kill the rest.
  4. Progressive disclosure. Start with Chat + Settings (3 sections). Everything else appears when needed.
  5. Celebrate success. After setup, after pairing, after first chat — show a "You did it!" moment. Humans need closure.
  6. Hide power, not remove it. Advanced users still get all 22 config sections, 48 env vars, raw terminal. They just go through a gate.
  7. Every destructive action needs a confirmation. Disconnect, delete, revoke, remove — all need "Are you sure?" with a brief delay.
  8. Test with actual grandmas. Not hypothetical ones. Sit a real non-technical person down and watch them struggle. Fix what they trip on.
